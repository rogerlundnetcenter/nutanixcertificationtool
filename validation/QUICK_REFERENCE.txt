CRITICAL ACRONYMS & SERVICES
──────────────────────────────────────────────────────────────────────────────

DSF Services:
  • Stargate      = I/O storage operations (port 2009)
  • Curator       = Garbage collection, data compaction (port 2010)
  • Cerebro       = Replication & disaster recovery
  • Medusa        = Distributed metadata store (Cassandra)
  • Genesis       = Service monitoring & auto-restart
  • Zeus          = Cluster config database (Paxos consensus)
  • Prism         = Management console (port 9440)

Data Structures:
  • OpLog         = Write coalescing buffer (SSD, ~6GB)
  • Unified Cache = Dedup fingerprint cache (read acceleration)
  • Extent Store  = Primary data storage (SSDs/HDDs)
  • Extent Groups = 1MB logical units (managed by Stargate)

Storage Features:
  • RF2/RF3       = Replication Factor (full copies)
  • EC-X          = Erasure Coding (space-efficient, CPU-heavy)
  • ILM           = Information Lifecycle Management (auto-tiering)
  • Shadow Clones = Local read-only copies for boot storms

REST API v3 ESSENTIALS
──────────────────────────────────────────────────────────────────────────────

Design Principle: INTENT-BASED (Spec vs Status)

Auth:       HTTP Basic Auth (base64-encoded username:password)
Rate Limit: 429 = exceeds rate; implement exponential backoff

VM Operations:
  CREATE:   POST /api/nutanix/v3/vms {body: spec}
  UPDATE:   PUT /api/nutanix/v3/vms/{uuid} {body: spec, metadata}
  LIST:     POST /api/nutanix/v3/vms/list {body: filter criteria}
  STATUS:   GET /api/nutanix/v3/tasks/{task_uuid}

Cluster:
  LIST:     POST /api/nutanix/v3/clusters/list
  HEALTH:   POST /api/nutanix/v3/clusters/list (check status field)

Categories (for Flow):
  CREATE KEY:   PUT /api/nutanix/v3/categories/{key}
  ADD VALUE:    PUT /api/nutanix/v3/categories/{key}/{value}

Bulk Operations:
  BATCH:        POST /api/nutanix/v3/batch (multiple operations)

Error Codes:
  409 Conflict  = spec_version outdated (retry with fresh GET)
  429 Too Many  = rate limited (backoff + retry)

STORAGE CONFIGURATION REFERENCE
──────────────────────────────────────────────────────────────────────────────

Feature              VDI Workload    OLTP Database   Cold Archive
─────────────────────────────────────────────────────────────────────────────
Replication Factor   RF2             RF3             RF2
Compression          Post-process    Post-process    Inline (LZ4)
Deduplication        YES (Fingerp.)  NO              YES
Erasure Coding       NO              NO              YES (EC-X)
Shadow Clones        YES             NO              NO
QoS (per-VM)         Throttle light  No limit        Irrelevant

Compression Delay Settings:
  • delay=0           → Inline compression (immediate, LZ4, CPU+latency)
  • delay=3600        → Post-process (1 hour delay, async, lower overhead)

OpLog Monitoring:
  • High util (90%+)  → Write ingestion exceeds destage rate
  • Normal (<60%)     → Healthy write performance
  • Check at:         http://CVM_IP:2009

NETWORKING TOPOLOGY
──────────────────────────────────────────────────────────────────────────────

Bond Modes (AHV):
  active-backup   → One active, others standby (no load balancing)
  balance-slb     → Hash-based LB, switch-independent (recommended)
  LACP (802.3ad)  → Requires physical switch LAG config (highest throughput)

OVS Hierarchy:
  Physical NICs → Bond → Bridge (br0) → Port → VM vNIC
                                    ↓
                            Trunk to Physical Switch

Jumbo Frames (MTU 9000):
  ✓ Must configure end-to-end:
    1. AHV host vSwitch uplinks
    2. Physical switch trunk port
    3. CVM network interfaces
  ✗ Mismatch causes fragmentation/drops

VPC & Flow:
  • VPC            → Overlay network (Geneve encapsulation)
  • Floating IP    → NAT for external access
  • VPN Gateway    → Tunnel to on-premises
  • Categories     → VM grouping (AppTier:Web, AppTier:DB, etc.)

Flow Policies (Priority Order):
  1. Quarantine   → Highest: Strict or Forensic mode
  2. Isolation    → Block all traffic between categories
  3. Application  → Allow specific flows on specific ports
  4. Default      → Lowest: Baseline deny-all

SECURITY & ENCRYPTION
──────────────────────────────────────────────────────────────────────────────

Data-at-Rest Encryption:
  • Algorithm:        AES-256 (software-based)
  • Requirement:      Key Management Server (KMS) configured
  • Protocol:         KMIP for external KMS (SafeNet, Vormetric, IBM)
  • Configuration:    Per-container in Prism

Cluster Lockdown:
  • Path:             Prism Element > Settings > Cluster Lockdown
  • Enforces:         Key-based SSH only (no password auth)
  • Benefit:          Hardens CVM access across cluster

STIG Hardening:
  • Tool:             SCMA (Security Configuration Management Automation)
  • Function:         Validates & remediates STIG compliance
  • Execution:        Automated, scheduled checks

SNMP v3 (vs v2c):
  • v2c:              Community string only
  • v3:               USM + authentication (MD5/SHA) + privacy (DES/AES)
  • Use Case:         Nagios, SolarWinds, Splunk integration

SSL Certificates:
  • Default:          Self-signed
  • Replace via:      ncli ssl-certificate replace
  • Includes:         Certificate + Private Key + Chain

COMMON GOTCHAS FOR EXAM
──────────────────────────────────────────────────────────────────────────────

❌ Mistake                      ✅ Correct Answer
──────────────────────────────────────────────────────────────────────────────
Using REST GET for v3 lists     Use POST (intentional design!)
Forgetting spec_version in PUT  Always include metadata.spec_version
Confusing ncli with acli        ncli=cluster, acli=AHV VMs
Mixing Stargate with Curator    Stargate=I/O, Curator=GC
Inline compression w/random WR  Use post-process (delay>0)
Erasure Coding for hot data     Use for cold/read-heavy only
Manual /etc/ssh edits           Use Prism Cluster Lockdown
Ignoring bond mode on vSwitch   balance-slb is switch-independent
RF3 everywhere for safety       Writes 3x = performance cost!
Shadow Clones on write-heavy    Trigger on reads only, killed by writes
Forgetting OVS end-to-end MTU   Must match at all layers
Quarantine policy order         Highest priority (blocks all)
ILM on all-SSD clusters         No effect (needs both tiers)
ncli instead of genesis         For single services, use genesis
Random port numbers in answers  Memorize: 2009, 2010, 9440

EXAM STRATEGY
──────────────────────────────────────────────────────────────────────────────

1. COMMAND QUESTIONS:
   → Eliminate obviously wrong paths (non-existent tools)
   → Recall ncli vs acli distinction
   → Remember: Port numbers are frequently wrong answers

2. API QUESTIONS:
   → Think "intent-based" = different from REST convention
   → v3 uses POST for lists (unusual but correct)
   → Always include spec_version in PUT requests

3. FEATURE QUESTIONS:
   → Match workload type to storage config
   → Remember EC-X trade-offs (CPU overhead for space savings)
   → Compression: delay=0 is inline, delay>0 is post-process

4. SECURITY QUESTIONS:
   → STIG uses SCMA tool (not manual checks)
   → Flow priorities: Quarantine > Isolation > App > Default
   → Encryption needs KMS (not just on nodes)

5. TROUBLESHOOTING QUESTIONS:
   → Always check Stargate:2009 first (I/O metrics)
   → Then CVM health (CPU/mem/services)
   → Look for data locality issues (cross-node I/O)

6. TIME MANAGEMENT:
   → Read all answer options fully
   → Watch for trick answers (technically true but not best)
   → If unsure, eliminate service name mismatches first
